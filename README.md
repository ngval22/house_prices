
# House Prices
Advanced Regression Techniques - [Kaggle Competition]("https://www.kaggle.com/competitions/house-prices-advanced-regression-techniques/overview")
## შესავალი
-   კონკურსის მიზანია დააპროგნოზო, თუ რა ფასად გაიყიდება სახლი, როდესაც ამ სახლის შესახებ გარკვეული მახასიათებლები გაქვს მოცემული. მონაცემები ტაბულარულია. Training Set-ში მოცემულია წარსულში გაყიდული სახლების 1460 ნიმუში. თითოეულ ნიმუშში გვაქვს Features (სახლის აღწერილობა, სართული, კვადრატულობა, ქუჩის ტიპი და ბევრი სხვა დეტალი, რომლებიც თითქმის სრულად აღწერს სახლს) და Target (SalePrice). 
ჩვენმა დატრენინგებულმა მოდელმა ახალ შემოსულ მახასიათებლების მიხედვით მაქსიმალურად სწორად უნდა "გამოიცნოს" თუ რამდენად გაიყიდება ეს სახლი.

- პრობლემის გადასაჭრელად ჯერ გავაანალიზე მოცემული დატასეტი, რაზე დაყრდნობითაც ავაგე პრეპროცესინგის პაიპლაინი, რომელიც მოიცავდა feature engineering-ის სხვადასხვა მიდგომებს: ordinal encoding, one hot encoding. შემდეგ feature selection სთვის გამოვიყენე ჯერ ვარიაციის, შემდეგ კი კორელაციის ფილტრები, რომ სვეტები გადამეგდო და ამით ხმაურიც შემემცირებინა მოდელისთვის და უფრო ეფექტური ყოფილიყო ტრენინგი. ტრენინგისთვის გამოვიყენე 3 მთავარი მოდელი: Linear Regression, Decision Tree და GradientBoosting. თითოეულისთვის შევადგინე ჰიპერპარამეტრის გრიდი, რომლითაც პარამეტრების რამდენიმე კომბინაციით გავუშვი ტრენინგი.  შედეგი RMSE, MAE, R^2^ მეტრიკებით გავაანალიზე.

## Feature Engineering
- ის ცვლადები, რომლებიც ძალიან სუსტ კორელაციაში (<0.05) არიან target-თან ან ბევრი N/A პროცენტულობა ჰქონდათ (>70) ბრმად არ დამიდროპავს.
 მაღალი nan მაჩვენებელი ყოველთვის არ ნიშნავს, რომ ამ მონაცემის შესახებ ინფორმაცია არ გვაქვს. ჩვენ შემთხვევაში ზოგიერთ ცვლადში ეს ნიშნავდა, რომ უბრალოდ არ აქვს სახლს ის ატრიბუტი რომელსაც ეს ცვლადი აღწერს. მაგალითად PoolQC აღნიშნავს აუზის ხარისხს. nan ამ შემთხვევაში აღნიშნავს რომ სახლს აუზი საერთოდ არ აქვს ეს კი ძალიან მნიშვნელოვანი სიგნალია სახლის ფასთან მიმართებაში და ცხადია არ უნდა დავდროპოთ. სხვა ასეთი ცვლადებია Alley, Fence, FireplaceQu. nan-ები ჩავანაცვლე სტრინგით none, სანამ კატეგორიულს რიცხვითად გარდავქმნიდი.
![Top N/A Values](https://github.com/ngval22/house_prices/blob/main/assets/Screenshot%202026-04-14%20at%2016.14.01.png)

- კატეგორიული ცვლადებისთვის გამოვიყენებ One-Hot-Encoding, თუმცა ზოგიერთი ცვლადი რაღაცის ხარისხის/იერარქიის განმსაზღვრელი იყო და უბრალო ohe-თი ამ ინფორმაციას დავკარგავდით. დამჭირდა სპეციალური ordered ენქოდინგი. მაგ: FireplaceQu: 'None', 'Po', 'Fa', 'TA', 'Gd', 'Ex'. ამ მეთოდით მნიშვნელობებს 0-დან 5-მდე რიცხვი შეესაბამება და ხარისხზე ინფორმაციას შევინარჩუნებთ.
- შემოვიტანე ახალი ცვლადი ჯამური ფართობის განმსაზღვრელი: TotalSF=TotalBsmtSF+1stFlrSF+2ndFlrSF. იგივენაირად: TotalBathrooms, TotalPorchSF. ვდროპავ ცვლადებს, რომლებიც ამ ახალი ცვლადების შესაქმნელად გამოვიყენე, ვინაიდან დამატებით იმდენ ინფორმაციას არ მისცემდა მოდელს, რომ მათი შენარჩუნება ღირდეს სვეტების სიმრავლის გათვალისწინებით.

## Feature Selection

- Variance Thresholdად გამოვიყენე 0.95*(1−0.95)  ზღვარი. ეს მოაშორებს თითქმის უცვლელ ან დაბალი ვარიაციის მქონე სვეტებს, სადაც ერთი მნიშვნელობა დომინირებს 95%-ზე მეტ შემთხვევაში. ასეთი მახასიათებლები მცირე ინფორმაციას შეიცავს და სავარაუდოა რომ დამატებითი სიგნალის ნაცვლად უბრალოდ noise დაამატოს.
- Correlation Filter
ვარიაციის ეტაპის შემდეგ, სვეტები გაიფილტრა თითოეული დარჩენილი მახასიათებლის კორელაციის მიხედვით თარგეთთან (SalePrice). შენარჩუნდა მხოლოდ |corr| > 0.1 მახასიათებლები. ამის შედეგად, მოდელებს გადაეცემათ მხოლოდ ის ცვლადები, რომლებსაც სამიზნესთან მინიმალური კავშირი მაინც აქვთ.

### Linear Regression

მოდელი გაეშვა ოთხ ალფა მნიშვნელობაზე (0.1, 1.0, 10.0, 100.0) როგორც Ridge-სთვის, ასევე Lasso-სთვის, პლიუს უბრალო LinearRegression საბაზისო მოდელი გავუშვი რეგულარიზაციის გარეშე.
 ![Results](https://github.com/ngval22/house_prices/blob/main/assets/Screenshot%202026-04-16%20at%2012.02.26.png)
ფაქტობრიად ყველა მოდელი, საბაზისო LinearRegression და Lasso/Ridge რეგულარიზაციის სხვადასხვა ჰიპერპარამეტრით მიდგომის მიუხედავად დაახლოებით ერთნაირ train R^2^ (0.8459) da validation R^2^(0.792)-ს აჩვენებს. სხვაობა ტრეინსა და ტესტს შორის შესამჩნევია, რაც ნიშნავს რომ რაღაც დონის ოვერფიტი გვაქვს. იგივეს აჩვენებს სხვაობა ტრინისა და ვალიდაციის RMSE მეტრიკებს შორის. შედარებით კარგი შედეგი აქვს Ridge alpha=100 ვარიანტს.

ამ შედეგის სავარაუდო მიზეზი არის პრეპროცესინგი. ჩემი პრეპროცესინგის შემდეგ საბოლოოდ მაინც რჩება 71 სვეტი (one hot encoding-ით 200-მდე ადის და შემდეგ ვარიაციით და კორელაციის ფილტრით ჩამომყავს 71-მდე). სავარაუდოდ არ არის პრეპროცესინგი საკმარისი და მაინც ბევრი ხმაური მიყვება დატასეტს გაფილტვრის შემდეგაც.

### Decision Tree

grid ავაგე 3 ჰიპერპარამეტრის გათვალისწინებით: max_depth (3, 5, 10, არცერთი), min_samples_split (2, 5, 10) და min_samples_leaf (1, 2, 4, 5). კომბინაციები ავარჩიე bias-variance სპექტრის სრულად დასაფარად, ანუ დავინახავთ იმ შემთხვევას როცა მოდელი დი ვარიაციაში წავა და იმასაც, როცა დიდი bias ექნება.
![Results](https://github.com/ngval22/house_prices/blob/main/assets/Screenshot%202026-04-16%20at%2012.04.12.png)
depthNone + split2 + leaf1: Train R^2^ = 1.000, train RMSE = $0. როგორც მოსალოდნელი იყო მოდელმა ყველაფერი დაიზეპირა. რადგან ხეს სიღრმე არ შევუზღუდეთ მოდელმა უბრალოდ დატასეტის თითოეული სახლისთვის უნიკალური გზა გაიკვალა. ბუნებრივია, რომ ამ მოდელს ვერაფერში გამოვიყენებთ. სიღრმე როცა 10-მდე შევუზღუდეთ, მაინც ძალიან დიდია ოვერფიტი მოგვცა იმავე მიზეზის გამო.

depth3 მოდელები ოვერფიტის პრობლემას აგვარებენ, თუმცა კომპლექსურობა აკლიათ, ვინაიდან ყოველ ლეველზე მხოლოდ 3 "კი ან არა" შეკითხვის დასმა შეუძლიათ სახლთან დაკავშირებით, რაც საკმარისი არ არის რომ დატასეტის კომპლექსურობა დაიჭიროს. შედეგად მოდელი ხდება biased. პერფორმანსი ფუჭდება: ვალიდაციის R^2^ ~ 0.685 და ტესტის RMSE ~45000.

depth5 + split10 + leaf4 გვაძლევს საუკეთესო R^2^ ~ 0.805 და შედარებით მცირე ოვერფიტს, თუმცა, როგორც ტაბულარულ დატასეტზე მოსალოდნელი იყო, საუკეთესო decision tree მოდელიც კი ვერ უწევს კონკურენცია წრფივ რეგრესიას. შედეგის გასაუმჯობესებლად საჭიროა რაიმე სახის Boosting, ამიტომ საბოლოო მიდგომად GradientBoosting ავარჩიე.

### GradientBoosting

ამ შემთხვევაში ჰიპერპარამეტრებია: n_estimators (100–500), learning_rate (0.05, 0.1), and max_depth (3–5). 
![Results](https://github.com/ngval22/house_prices/blob/main/assets/Screenshot%202026-04-16%20at%2012.05.29.png)
validation R^2^ ამ შემთხვევაში კონსისტენტურად ბევრად მაღალია decision tree თან შედარებით, რაც ნიშნავს რომ ბუსტინგი თავის საქმეს აკეთებს, ბევრი მარტივი ხით, სადაც ყოველი აუმჯობესებს წინა ხეების შედეგებს. თუმცა, ტრეინსა და ვალიდაციას შორის მაინც არის გარკვეული სხვაობა, რაც ოვერფიტინგზე მიგვანიშნებს.

მეტრიკების მიხედვით საუკეთესო მოდელია: n200, lr=0.1, depth3. ვალიდაციის R^2^= ~0.866, ტესტ R^2^= ~0.886. მცირე ოვერფიტი, როგორც უკვე ვთქვი აქაც გვაქვს, მაგრამ დაბალი სიღრმის ხეებმა მაინც ბევრად უკეთ განაზოგადა ვიდრე მაღალმა.

max_depth - თითოეული სუსტი ხის მაქსიმალური სიღრმე
